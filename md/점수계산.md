# 🏆 Gig-Market Trust Score 산정 시스템 설계 제안

## 핵심 철학: "행동 기반 신뢰 점수 (Behavior-Driven Trust)"

단순 별점 평균이 아닌, **실제 서비스 행동 데이터**를 종합하여 조작이 어렵고 신뢰도 높은 점수를 산출하는 것이 목표입니다.

## 📊 Trust Score 구성 요소 (총 1,000점)

### 1️⃣ 서비스 완성도 (Service Quality) — **350점 최대**

| 지표 | 배점 | 측정 방법 |
|------|------|-----------|
| 고객 별점 평균 | 200점 | 누적 리뷰 가중평균 (최근일수록 가중치 높음) |
| 서비스 완료율 | 100점 | 완료건수 / 수락건수 |
| 재고용률 (Repeat Hire) | 50점 | 동일 고객이 재의뢰한 비율 |

> **포인트:** 재고용률은 조작이 거의 불가능한 강력한 품질 지표입니다.

## ⚙️ 서버 알고리즘 설계 핵심 아이디어

### 💡 아이디어 1: 시간 감쇠 가중치 (Time Decay Weight)

오래된 리뷰보다 **최근 6개월 행동**에 더 높은 가중치를 부여합니다.

```typescript
// NestJS 서버 로직 예시
function calculateWeightedRating(reviews: Review[]): number {
  const now = Date.now();
  const SIX_MONTHS = 180 * 24 * 60 * 60 * 1000;
  
  let weightedSum = 0;
  let totalWeight = 0;
  
  reviews.forEach(review => {
    const age = now - new Date(review.created_at).getTime();
    // 최근일수록 weight = 1.0, 6개월 전 → 0.5
    const weight = Math.max(0.5, 1 - (age / SIX_MONTHS) * 0.5);
    weightedSum += review.rating * weight;
    totalWeight += weight;
  });
  
  return totalWeight > 0 ? weightedSum / totalWeight : 0;
}
```

### 💡 아이디어 2: 연속 성과 보너스 / 페널티 (Streak System)

```
✅ 5연속 5-star → +10 보너스 포인트
✅ 10연속 무분쟁 완료 → +20 포인트
❌ 3연속 낮은 평점(3점 이하) → -15 포인트 + 관리자 경고
❌ 연속 무단취소 2회 → -30 포인트 + 계정 일시 제한
```

### 💡 아이디어 3: 카테고리별 분리 점수 (Category-Specific Score)

같은 Provider라도 **"에어컨 청소"와 "전기 배선"은 별도 점수**로 관리합니다. 이를 통해 전문성 신뢰도를 구체화합니다.

```sql
-- PostgreSQL 스키마 예시
CREATE TABLE provider_category_scores (
  provider_id     UUID REFERENCES providers(id),
  category_id     UUID REFERENCES service_categories(id),
  score           INTEGER DEFAULT 500,
  total_jobs      INTEGER DEFAULT 0,
  avg_rating      DECIMAL(3,2),
  updated_at      TIMESTAMP,
  PRIMARY KEY (provider_id, category_id)
);
```

### 💡 아이디어 4: 리뷰 신뢰도 필터 (Review Credibility Filter)

부정 리뷰 조작 방지를 위한 서버 검증 로직입니다.

```
반영 조건 (모두 충족 시에만 Trust Score 반영):
✅ 실제 완료된 Contract 건에 한해서만 리뷰 가능
✅ 고객 계정 생성 후 7일 이상 경과
✅ 고객이 2건 이상 거래 이력 보유
✅ 동일 고객이 동일 Provider에 1건 이상 5-star 부여 시 자동 플래깅
```

### 💡 아이디어 5: 자동 티어 강등 보호 (Tier Drop Protection)

티어가 낮아질 경우 Provider 이탈을 막기 위한 완충 장치입니다.

```
Elite → Pro 강등 조건:
- Trust Score가 750 미만으로 30일 연속 유지될 때만 강등
- 강등 7일 전 사전 알림 발송 (Push + SMS)
- 30일 내 750 회복 시 강등 취소
```

## 🗄️ PostgreSQL 핵심 테이블 구조

```sql
CREATE TABLE provider_trust_scores (
  id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  provider_id           UUID NOT NULL REFERENCES providers(id),
  
  -- 점수 구성 요소
  quality_score         INTEGER DEFAULT 0,     -- /350
  reliability_score     INTEGER DEFAULT 0,     -- /250
  integrity_score       INTEGER DEFAULT 0,     -- /200
  activity_score        INTEGER DEFAULT 0,     -- /100
  verification_score    INTEGER DEFAULT 0,     -- /100
  
  -- 종합 점수
  total_score           INTEGER GENERATED ALWAYS AS (
                          quality_score + reliability_score + 
                          integrity_score + activity_score + 
                          verification_score
                        ) STORED,
  
  -- 티어
  tier                  VARCHAR(20) DEFAULT 'New',
  -- 'New' | 'Basic' | 'Verified' | 'Pro' | 'Elite'
  
  -- 메타
  last_calculated_at    TIMESTAMP DEFAULT NOW(),
  score_version         INTEGER DEFAULT 1,
  
  CONSTRAINT score_range CHECK (total_score BETWEEN 0 AND 1000)
);

-- 변동 이력 테이블
CREATE TABLE trust_score_history (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  provider_id     UUID NOT NULL REFERENCES providers(id),
  score_before    INTEGER NOT NULL,
  score_after     INTEGER NOT NULL,
  change_amount   INTEGER NOT NULL,
  change_reason   VARCHAR(100) NOT NULL,
  -- 예: 'REVIEW_5STAR', 'DISPUTE_RAISED', 'STREAK_BONUS'
  source_id       UUID,  -- contract_id 또는 dispute_id 참조
  created_at      TIMESTAMP DEFAULT NOW()
);
```

## 🔄 Trust Score 재계산 트리거 (NestJS)

```typescript
// 재계산이 필요한 이벤트들
@OnEvent('contract.completed')
@OnEvent('review.submitted')
@OnEvent('dispute.resolved')
@OnEvent('provider.verified')
async recalculateTrustScore(providerId: string) {
  // 배치가 아닌 이벤트 기반으로 즉시 반영
  await this.trustScoreService.recalculate(providerId);
}
```

재계산은 **이벤트 기반(Event-Driven)**으로 처리하여 DB 부하 없이 실시간에 가깝게 반영하는 것을 권장합니다.
